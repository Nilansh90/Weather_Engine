I think this is a much stronger direction. It shifts the dashboard from **"system monitoring"** to
**"model evaluation"**, which is exactly what someone evaluating an ML weather engine wants to see.

## Final Dashboard Strategy

---

### 1. KPI Cards (Top)

Keep only:

* Temperature MAE
* Pressure MAE
* Moisture MAE *(combined dew point + humidity, as you currently calculate)*
* Rain Accuracy (%)
* Cities Processed
* Forecasts Generated

**Remove:**

* ❌ Average Runtime
* ❌ Weather Code Accuracy

---

### 2. Overall Model Performance (Plotly)

These should use the last **15 days** of data, or **all available days if fewer than 15 exist**.

#### Temperature MAE Trend

* X-axis: Forecast Date
* Y-axis: Mean Temperature MAE (averaged across all cities)

#### Pressure MAE Trend

* X-axis: Forecast Date
* Y-axis: Mean Pressure MAE

#### Dew Point MAE Trend

* X-axis: Forecast Date
* Y-axis: Mean Dew Point MAE

#### Humidity MAE Trend

* X-axis: Forecast Date
* Y-axis: Mean Humidity MAE

#### Rain Classification Accuracy

* Daily bar chart
* Y-axis: Accuracy (%)
* Average across all cities for each day

**Remove:**

* ❌ Weather Code Accuracy chart

---

### 3. Forecast vs Actual

These should compare predictions against observations.

#### Temperature

Two-line Plotly graph

```
Prediction
────────────

Actual
────────────
```

Last 15 days.

---

#### Pressure

Same format.

---

### 4. City Performance Table

Keep this section.

Instead of placeholder values, compute directly from the `errors` table.

Suggested columns:

| City | Temperature MAE | Pressure MAE | Dew Point MAE | Humidity MAE | Rain Accuracy | Forecasts Evaluated |
| ---- | --------------- | ------------ | ------------- | ------------ | ------------- | ------------------- |

This becomes a leaderboard showing which cities are easiest or hardest for the model.

---

## Remove Entirely

* ❌ Pipeline Health
* ❌ Engine/Model Health
* ❌ Pipeline Timeline
* ❌ City Coverage Map
* ❌ Latest Forecasts (already covered on the homepage)
* ❌ Weather Code Accuracy metrics and graphs
* ❌ Error Distribution charts (histograms/box plots)
* ❌ Runtime metrics

---

## Data Source Plan

Everything should come from the existing tables you already have:

### `errors`

Used for:

* Temperature MAE
* Pressure MAE
* Dew Point MAE
* Humidity MAE
* Rain Accuracy
* City-wise metrics
* Daily trend graphs

### `predictions`

Used for:

* Forecast values (Forecast vs Actual)

### `weather_data`

Used for:

* Actual observations (Forecast vs Actual)

### `metrics`

Used only for:

* Top KPI cards (overall summary values)
